# Holistic Review: Security & Crypto Posture

**Date:** 2026-07-20
**Ticket:** [Holistic review: security & crypto posture](https://github.com/mikmorg/tapectl/issues/4) (wayfinder map [#1](https://github.com/mikmorg/tapectl/issues/1))
**Method:** single security auditor swept `src/crypto/`, `src/volume/`, `src/staging/`, `src/tenant/`, `src/config.rs`, `src/main.rs`, `src/db/`, and `src/tape/ioctl.rs` against `tapectl-design-v4_0.md` (§2.1, §2.6, §2.16, §8.x), the four ADRs, and `CONTEXT.md`. Every finding is cited to a file:line read during the audit. Cryptographic property claims about the `age` format are verified against the C2SP age v1 specification and the `age` 0.11.2 source, cited inline with URLs. `cargo audit` was run (already installed at `/scratch/cargo/bin/cargo-audit`). Severity is weighted per the renovation intent: **heir-restore capability and key custody heaviest; stranger-facing metadata polish light.**
**Verdict scope:** static review plus dependency scan. The threat model taken as authoritative is the one in `CONTEXT.md` / the intent statement — one operator, tenants are *data classes* not adversaries, threats are device failure, site loss, bitrot over decades, and future-self/heir usability. Findings that assume an active tampering adversary are scored as defense-in-depth against that model and say so.

---

## Executive summary

The cryptographic core is sound and, in one important respect, **stronger than this audit's own commissioning premise assumed**. The prompt hypothesised that "an adversary with the tape can rewrite mini-index + checksums consistently." That is not how the volume is built: the plaintext mini-index carries **no checksums at all** (`layout.rs:601-604` emits only position, type label, and byte size), and the authoritative per-slice SHA-256 values live inside the tenant/operator **envelope**, which is a tar archive that is age-encrypted *and therefore authenticated* before it touches tape (`write.rs:307-308`). An heir's expected hashes come from an authenticated source, so slice substitution, reordering, and truncation **fail safe** rather than silently corrupting the restore. This is the headline: the integrity story is good, and it is good for a precise, nameable reason.

That said, the properties the intent statement weighs heaviest — **key custody and heir-restore** — carry the real exposure, and it is not in the cipher. It is that every private key lives unencrypted in one directory on one machine with no escrow, no passphrase, and no split (S1); and that the emergency-restore artifact the heir runs blind, `RESTORE.sh`, evaluates values parsed from attacker-writable plaintext tape zones inside bash arithmetic, which is a command-injection primitive (S2). Neither is a flaw in `age`; both are in the custody and recovery envelope around it.

### Theme 1 — Correcting the integrity premise (the core result)

`age` provides **intra-file** authenticated encryption: each 64 KiB chunk is sealed with ChaCha20-Poly1305, the final chunk carries a distinguishing nonce flag so truncation/appending is detected, and the header is bound by HMAC-SHA-256 over its entire contents (age v1 spec, C2SP; retrieved 2026-07-20, https://github.com/C2SP/C2SP/blob/main/age.md). What `age` does **not** provide is any binding between a ciphertext and its filename, tape position, or neighbours — associated data is empty, so two validly-encrypted files can be swapped or reordered by the format's own guarantees. In tapectl that **inter-file** integrity is supplied by the authenticated manifest, not by `age`: the heir reads `sha256_encrypted` from the decrypted (hence authenticated) envelope and checks each slice before use (`layout.rs:637-643`, consumed by generated `RESTORE.sh` and by `restore.rs:102-133`). Because a stranger holding the tape cannot forge a manifest (they lack a recipient key) and cannot rewrite the mini-index checksums (there are none), the residual authenticity gap is narrow and specific: **whole-tape rollback** — swapping an entire sealed tape for an older authentic one of the same label restores stale-but-genuine data with no freshness anchor the heir can check (S10).

### Theme 2 — Key custody is the heaviest real risk (1 HIGH)

All tenant and operator private keys are generated to and read from `~/.tapectl/keys/*.age.key` as raw, unencrypted X25519 identity files (`tenant/mod.rs:24-62`, `crypto/keys.rs:43-56`). There is no passphrase wrapping, no escrow, no Shamir split, and the designed paper-backup path (`key export --qr`) is unimplemented (prior audit shell-LOW). For a decades/heir horizon the dominant failure is not cryptanalysis — it is the heir never finding a key, or a single-site loss destroying keys and (if co-located) tapes together. This is the in-model **site-loss** threat landing squarely on the key layer, and it feeds ticket #19 (escrow) — named here, not decided.

### Theme 3 — The blind-heir recovery script is a tamper-to-RCE surface (1 MED, defense-in-depth)

`RESTORE.sh` (generated in `layout.rs:167-545`) parses `num_envelopes`, `first_envelope`, and friends out of the plaintext ID thunk / mini-index — tape zones any holder of the physical cartridge can rewrite — and feeds them into bash arithmetic such as `$((FIRST_ENV + NUM_ENV - 1))`. Bash arithmetic expansion performs command substitution and array-subscript evaluation, so a crafted value (`$(cmd)` or `a[$(cmd)]`, space-free) yields code execution on the heir's machine, before any envelope is even decrypted. Under the stated threat model (no active adversary) this is defense-in-depth, hence MED; it escalates to HIGH the moment untrusted-custody enters scope. It is the **only** plaintext-zone tamper that results in RCE rather than a safe abort, and it fires via `--info` independently of prior-audit H1.

### Theme 4 — Plaintext metadata custody, on disk and on tape (2 MED, 1 LOW)

The tape design's "zero content metadata in plaintext" rule is upheld on tape (no names, filenames, hashes, or ownership leak), but the same metadata sits **plaintext on disk**: the SQLite catalog, receipts, and dar catalogs (all mode 0644) hold filenames, paths, sizes, and SHA-256s, in a home whose directories are created 0755 (`config.rs:386-397`) — world-readable on any multi-user host (S4). Separately, `db backup` silently copies the entire private-key directory to an operator-chosen `<dest>.keys/` (`operations.rs:782-799`) — surprising secret propagation that also happens to be the *only* key-escrow path that exists (S3).

### Theme 5 — Supply chain is clean except a DoS in dead code (1 MED)

`cargo audit` flags two 7.5-severity DoS advisories, both in `quick-xml` 0.39.2 (RUSTSEC-2026-0194/0195) — reachable only through `dar/catalog_xml.rs`, which the prior audit found is entirely dead code. `age` is pinned at 0.11.2 as the design intends; `rusqlite` 0.39 bundles SQLite **3.51.3** (verified, current, no known CVEs); no git dependencies; the two `unsafe` blocks are standard, correctly-guarded ioctl FFI.

## Detailed findings

### [HIGH] S1 — Private keys unencrypted, single-copy, single-site; no escrow

- **Where:** `src/tenant/mod.rs:24-62` (auto-generates `primary`+`backup` keypairs to disk on tenant/operator creation), `src/crypto/keys.rs:43-56` (`save_secret_key` writes a raw `AGE-SECRET-KEY-...` line, mode 0600), `src/crypto/keys.rs:102-127` (`load_all_identities` reads them straight back)
- **Design ref:** §2.1 ("operator generates all keypairs and holds all private keys"), §2.16 ("operator generates all keypairs, distributes private keys… `key export ALIAS [--qr]` for paper backup"); CONTEXT.md intent — site loss and heir usability
- **What:** Every tenant's and the operator's decryption capability rests on plaintext X25519 identity files in one directory on one machine. There is no passphrase/scrypt wrapping (the files are protected only by 0600 mode), no escrow, no secret-sharing, and no enforced off-host backup. The design's only backup affordance, `key export --qr`, is unimplemented (prior audit shell-LOW). The consequences map directly onto the two heaviest in-model threats: (1) **site loss** — a fire that takes the machine takes every key with it, and if tapes are co-located, the tapes become permanently unrecoverable *even though they are physically intact and self-describing*; (2) **heir usability** — the tapes carry excellent instructions on *how* to decrypt (system guide, RESTORE.sh, RECOVERY.md) but the one input those instructions require, the key, has no durable, discoverable, operator-independent backup. `age`'s per-file security is irrelevant if the key is gone. Note this is the custody counterpart to prior audit **H13** (a second `key rotate` strands a tenant keyless): H13 is a code bug that destroys a key's *usability*; S1 is the systemic absence of any redundancy for keys that are, by design, irreplaceable.
- **Remediation:** Do not decide the mechanism here — this feeds ticket #19. Options to weigh: passphrase-wrapped identity files (`age` supports scrypt recipients) so a stolen disk is not a stolen key; implement `key export --qr` for paper escrow; document a mandatory "keys live in a different failure domain than tapes" operating procedure; consider Shamir split for the operator key. At minimum, surface the risk at `init` and in the audit output.

### [MED] S2 — `RESTORE.sh` executes attacker-writable tape values in bash arithmetic (tamper → RCE)

- **Where:** `src/volume/layout.rs:167-545` (the generated script). The injectable sites are the arithmetic expansions over tape-derived variables: `$((FIRST_ENV + NUM_ENV - 1))` in `do_info` (generated ~line 301), `do_find_envelope` (~323), and `do_restore` (~379). `FIRST_ENV`/`NUM_ENV`/`OP_ENV`/`OP_BAK` are extracted by `toml_val` from the plaintext ID thunk (`layout.rs:264-281`, generated).
- **Design ref:** §8.1/§8.6 (ID thunk and mini-index are plaintext, structural), §2.6 (Files 0-2 + mini-index plaintext); ADR-0001 (tape is authoritative for itself — but nothing authenticates the plaintext zones)
- **What:** The ID thunk and mini-index are, by design, unauthenticated plaintext that any holder of the physical tape can rewrite. `RESTORE.sh` parses integers out of them and uses them in bash `$(( ))` arithmetic. Bash arithmetic expansion performs parameter expansion **and command substitution** on the expression text before evaluating it, and evaluates array subscripts — so a value like `$(payload)` or `a[$(payload)]` (no spaces required; `${IFS}` or brace tricks cover any that are) triggers command execution. A tampered tape whose ID thunk sets `num_envelopes = a[$(curl…|sh)]` runs that command the instant the heir types `./RESTORE.sh --info` — i.e. before any key handling, envelope decryption, or checksum check. The heir runs this script blind, by definition on an artifact whose provenance they may not control, possibly as root. Under the *stated* threat model (tenants are data classes, no active adversary) tape tamper is out of scope, so this is defense-in-depth — hence MED, not HIGH. But it is the single exception to the otherwise-clean "plaintext-zone tamper fails safe" property: every other tampered plaintext value (sizes, positions) leads to a `truncate`/`age`/`sha256` mismatch and a clean `die`, whereas this one leads to code execution. It also fires independently of prior-audit **H1** (the mini-index/envelope defect), because `--info` reaches the vulnerable arithmetic before it ever touches an envelope.
- **Remediation:** Validate every value parsed from the tape as a non-negative decimal integer before use (`case "$x" in *[!0-9]*) die ;; esac`), and prefer `expr`/`awk` or `[ "$a" -lt "$b" ]` comparisons over `$(( ))` on tape-derived data. Add a round-trip test that writes a hostile ID thunk and asserts `RESTORE.sh --info` refuses rather than executes.

### [MED] S3 — `db backup` silently exfiltrates all private keys to an operator-chosen path

- **Where:** `src/cli/operations.rs:782-799` (`db_backup` — after the SQLite backup it does `copy_dir_all(&paths.keys_dir, &keys_backup)` where `keys_backup = <dest>.keys`), `src/cli/operations.rs:859-871` (`copy_dir_all` — `create_dir_all(dst)` with default umask, then `fs::copy` per file)
- **Design ref:** §8.8 ("private keys are NEVER written to tape"), quality; CONTEXT.md custody
- **What:** `tapectl db backup --to /mnt/usb/backup.db` does the obvious thing (copies the catalog) and one non-obvious thing: it copies the **entire private-key directory** to `/mnt/usb/backup.db.keys/`. The `.key` files retain 0600 (Rust `fs::copy` copies permission bits), but the containing directory is created with the process umask (typically 0755), and — more importantly — the operator likely does not expect a "database backup" to also duplicate every tenant's decryption key onto removable, cloud-synced, or shared storage wherever `--to` points. This is genuinely double-edged: it is currently the *only* mechanism that gets keys off the primary host at all, so it partially mitigates S1 — but silently and undocumented, which is the wrong way to run a secret out of its containing failure domain.
- **Remediation:** Make the key copy explicit and opt-in (`--include-keys`), print a prominent warning naming the destination when keys are copied, and `create_dir_all` the target with mode 0700. Document it as an escrow affordance so it can be used deliberately (ties to S1/#19).

### [MED] S4 — On-disk custody: plaintext content-metadata is world-readable

- **Where:** `src/config.rs:386-397` (`ensure_dirs` — `create_dir_all` on home, keys, catalogs, receipts, logs with no explicit mode → 0777 & ~umask, typically 0755), `src/config.rs:345-353` (`Config::save` → `fs::write`, 0644), DB opened by `rusqlite` at default 0644, receipts written `fs::write` (`staging/mod.rs:318-319`, 0644), dar catalogs under `catalogs_dir` (0644)
- **Design ref:** §2.6 tenant-isolation rule ("NOTHING in plaintext reveals content, filenames, sizes, hashes, unit names, tenant names"); CONTEXT.md — the DB is a claims ledger
- **What:** The design goes to considerable lengths to keep content metadata off the tape in plaintext, but the same metadata is the *custody model* on disk and is not protected commensurately. `~/.tapectl/tapectl.db` (0644) holds every filename, path, size, mtime, and SHA-256 tapectl has ever seen (`files`, `manifest_entries`); receipts (0644) enumerate slice sizes and both plaintext and encrypted hashes; dar catalogs (0644) hold full file listings. On a multi-user host any local user can read all of it. The secret `.key` files themselves are correctly 0600 (S1 aside), and the DB legitimately *must* hold this metadata to function — so the finding is not "stop storing it" but "the plaintext-metadata custody boundary that the tape design enforces is silently dropped at the filesystem." State this explicitly as the model: **the DB, receipts, and catalogs must be protected like the plaintext content-metadata index the tape deliberately lacks.**
- **Remediation:** Create `~/.tapectl` and all subdirectories 0700 and set the DB/receipts/catalog files 0600 (an explicit `set_permissions` after creation, or a restrictive umask at `init`). Document that the home is a secret-bearing directory. On the documented single-user VM this is low-consequence today (hence MED not HIGH), but it is a one-line-per-path fix and the intent statement weighs custody heavily.

### [MED] S5 — Supply chain: known DoS advisories in `quick-xml`, reachable only via dead code

- **Where:** `Cargo.lock` (`quick-xml` 0.39.2, `age` 0.11.2, `rusqlite` 0.39.0, `libsqlite3-sys` 0.37.0); the only consumer of `quick-xml` is `src/dar/catalog_xml.rs`
- **Design ref:** §Dependencies; quality
- **What:** `cargo audit` (advisory DB loaded 2026-07-20) reports **2 vulnerabilities**, both `quick-xml` 0.39.2: RUSTSEC-2026-0195 (unbounded namespace-declaration allocation, memory-exhaustion DoS, CVSS 7.5, fixed ≥0.41.0) and RUSTSEC-2026-0194 (quadratic time on duplicate attribute names, 7.5, fixed ≥0.41.0). Both require feeding hostile XML to `NsReader`; tapectl's only XML path is `catalog_xml.rs`, which the prior audit found is entirely `#[allow(dead_code)]` and never called (the files catalog comes from tapectl's own directory walk). So the advisories are **not reachable** in the shipping binary — but the crate is still compiled in and would become live the moment anyone wires the module up. `cargo audit` additionally emits 6 lower-priority warnings assessed as non-issues here: `anyhow` 1.0.102 unsound `downcast_mut` (tapectl never downcasts), `rand` 0.9.2/0.8.5 unsound-with-custom-logger (no custom logger, no direct `rand::` use in `src/` — grep-verified), and unmaintained `proc-macro-error`/`proc-macro-error2`/`number_prefix` (build/format-only transitive deps). Positives: `age` is pinned 0.11.2 exactly as §2.16 requires; bundled SQLite is **3.51.3** (verified in `libsqlite3-sys-0.37.0/sqlite3/sqlite3.h`), current with no known CVEs; **zero git dependencies**.
- **Remediation:** Delete `src/dar/catalog_xml.rs` and drop `quick-xml` from `Cargo.toml` (it removes both advisories and a dead module in one move — see prior audit's dead-seam inventory), or, if the module is kept for a future feature, bump `quick-xml` to ≥0.41. Add `cargo audit` to the pre-commit checklist alongside clippy/fmt.

### [LOW] S6 — Secret key material lives in un-zeroized heap `String`s

- **Where:** `src/crypto/keys.rs:24` (`let secret_str = secret.to_string().expose_secret().to_string();` — copies the key out of `age`'s `Secret` wrapper into a plain `String`), propagated via `GeneratedKeypair.secret_key: String` and `read_secret_key` → `String` (`keys.rs:71-83`)
- **Design ref:** quality; CONTEXT.md single-operator
- **What:** `age`/`secrecy` hold identities in zeroizing wrappers, but tapectl immediately calls `.expose_secret().to_string()`, materialising the `AGE-SECRET-KEY-...` string on the heap with no zeroization on drop. The plaintext key can therefore linger in freed heap, be captured in a core dump, or be paged to swap. On the documented single-operator VM with no untrusted local users this is a low-consequence, accepted risk — but it is worth naming because it quietly discards the memory-hygiene the dependency already provides.
- **Remediation:** Keep key material in `secrecy::SecretString`/`Zeroizing<String>` end-to-end and pass identities without an intermediate plain `String`; if accepted as-is, record it as a conscious risk decision.

### [LOW] S7 — `RESTORE.sh` decrypts plaintext into a predictable `/tmp` path without `O_EXCL`

- **Where:** `src/volume/layout.rs:188-190` (generated: `WORK="${TMPDIR:-/tmp}/tapectl-restore-$$"; trap 'rm -rf "$WORK"' EXIT; mkdir -p "$WORK"`); decrypted dar slices land in `$WORK/dar/` (generated `do_restore`)
- **Design ref:** §8.3; quality. Contrast prior audit physical-LOW ("restore temp dir inside destination can leak decrypted slices on failure") — that finding is about the **Rust** `restore_unit` path (`volume/restore.rs:54`); this is a *distinct* issue in the **generated shell** path.
- **What:** The work directory is `/tmp/tapectl-restore-<pid>` — predictable (pid space is small) and created with `mkdir -p`, which succeeds on a pre-existing directory and sets no restrictive mode. A local attacker on the heir's recovery machine can pre-create or symlink that path to intercept or redirect the **decrypted** dar slices. The Rust restore paths are safer here (`restore_file` uses `tempfile::tempdir()`, 0700; `restore_unit` stages under the destination), so this is specific to the emergency script — the one most likely to be run on an unfamiliar machine.
- **Remediation:** Use `mktemp -d` for the work directory and `umask 077` at the top of the script.

### [LOW] S8 — Named-but-accepted plaintext structural leaks on tape

- **Where:** `src/volume/layout.rs:70-78` (ID-thunk `[layout]`: `data_start`, `data_end`, `num_envelopes`, `total_files`), `src/volume/layout.rs:601-604` (mini-index emits `position`/`type`/`size_bytes` per file), `src/volume/write.rs:158` (`num_tenant_envelopes` = count of distinct tenants)
- **Design ref:** §2.6 ("NOTHING in plaintext reveals content, ownership, filenames, sizes, hashes, unit names, tenant names"), §8.6 (mini-index "Contains NO filenames, tenant names, unit names, checksums")
- **What:** The design's rule is about *content* metadata, and the implementation upholds it precisely — no names, filenames, per-file content hashes, or ownership appear in plaintext. But a stranger holding the tape does learn a well-defined set of **structural** facts, and these should be on the record even though they are accepted by design: the volume label, creation timestamp, and tapectl version (ID thunk); the **number of tenants** sharing the tape (`num_envelopes` in both the ID thunk and the count of `tenant_envelope` entries in the mini-index); the **number of data slices** (`data_end − data_start + 1`, and `total_files`); and the **exact encrypted byte size of every slice and every envelope** (mini-index `size_bytes`, which §8.6 explicitly requires for block-padding trim). Slice sizes plus the design's "one unit's slices are contiguous" rule let an observer infer per-unit data volumes. None of this is a defect — the mini-index needs the sizes to function — but "zero content metadata" should not be read as "zero metadata."
- **Remediation:** Documentation only: state in the design and system guide exactly what structural metadata the plaintext zones expose, so the accepted leak is explicit rather than implied-absent.

### [LOW] S9 — Unsanitized tenant name interpolated into key file paths

- **Where:** `src/crypto/keys.rs:86-95` (`key_paths` → `keys_dir.join(format!("{tenant_name}-{alias}.age.key"))`), reached from `src/tenant/mod.rs:9-26` (`add_tenant` takes the name straight from the CLI with no validation) and `src/crypto/keys.rs:102-117` (`load_all_identities` matches by `starts_with("{tenant_name}-")`)
- **Design ref:** §2.1 (tenant naming); quality
- **What:** Tenant names are interpolated into filesystem paths without validation. A name containing `/` or `..` (e.g. `../../evil`) would write or read key files outside `keys/`. Tenant names are operator-supplied on a single-operator system, so exploitation requires the operator to attack themselves — hence LOW — but there is no defensive check anywhere, and the `starts_with` prefix match in `load_all_identities` is also fragile to naming that embeds the separator.
- **Remediation:** Validate tenant (and unit) names against a conservative charset (`[A-Za-z0-9._-]`, no leading `.`/`-`) at creation; reject path separators.

### [LOW] S10 — No volume-level freshness anchor: whole-tape rollback restores stale-but-authentic data

- **Where:** design §2.6/§8; the restore paths trust any sealed volume that decrypts and checksums (`src/volume/restore.rs:102-133`, generated `RESTORE.sh` `do_restore`)
- **Design ref:** §2.18 verification, ADR-0001 (tape authoritative for itself), ADR-0004 (evidence age warns, never gates)
- **What:** This is the precise residual of the Theme-1 integrity analysis. Per-slice substitution, reordering, and truncation all fail safe, because the expected `sha256_encrypted` comes from the age-authenticated envelope and a mismatch aborts. What no on-tape mechanism catches is **whole-object rollback**: an older, internally-consistent, correctly-sealed tape bearing the same label decrypts cleanly and passes every check, so an heir restoring from it silently gets a stale version. `age` cannot detect this (each file is independently authentic); only cross-volume catalog state — which an heir explicitly does not have — distinguishes "current" from "superseded." For the heir threat this is arguably acceptable (an authentic older backup is still their data), but it is the one authenticity property the self-describing tape does not carry, and it should be named rather than assumed away. It is distinct from bitrot detection, which `age`'s per-chunk Poly1305 does cover at read time (though prior audit **H6** notes the *staging-time* baseline can be clobbered before commit).
- **Remediation:** If freshness matters to the heir, bind a monotonic volume/version stamp and creation time into the envelope manifest (already partly present as `created_at`) and have RESTORE.sh surface it prominently ("this tape was sealed 2019-03-11; newer copies may exist"). A cryptographic anchor (operator-signed volume record) would be a larger design step — note for a future ADR, do not decide here.

## Verified clean

Each item below was opened and checked; the claim is a verified absence of a defect, not an unexamined area.

- **age encryption construction is correct.** `encrypt_data` (`src/staging/mod.rs:387-419`) builds `age::Encryptor::with_recipients` over all tenant + operator X25519 recipients, wraps a streaming writer, `write_all`s, and `finish`es — **binary format, no armor** (correct for tape; armor would inflate and add a text-parsing failure mode). Multi-recipient means any one recipient key decrypts, exactly as §2.16/§8 require. The whole-buffer nature of the call (plaintext held in RAM) is prior audit **H9**, an OOM/scale issue, not a crypto defect — the ciphertext is fully formed and authenticated before any byte is written.
- **age decryption trial-loop fails closed.** `restore_unit` (`src/volume/restore.rs:57-124`) loads every secret key for the tenant *and* the operator, hands all identities to `Decryptor::decrypt`, and propagates any error. A wrong key yields an `Err`, never silent garbage. `age` authenticates plaintext (per-chunk Poly1305), so a corrupted or substituted ciphertext fails rather than returning corrupt data — pinned by `tests/tenant_isolation.rs` (`cross_tenant_decrypt_is_rejected`, `tampered_ciphertext_fails_to_decrypt`) and consistent with the age v1 spec (empty AAD; final-chunk nonce flag prevents truncation; header HMAC-SHA-256 over the whole header — C2SP age spec, retrieved 2026-07-20).
- **Key generation uses a CSPRNG.** `x25519::Identity::generate()` seeds from `OsRng` (`StaticSecret::random_from_rng(OsRng)`), i.e. the OS entropy source via `getrandom` (age 0.11.2 source, https://docs.rs/age/0.11.2/src/age/x25519.rs.html, retrieved 2026-07-20). No custom or predictable RNG is involved.
- **Secret-key file mode is 0600.** `save_secret_key` opens with `OpenOptions…mode(0o600)` (`src/crypto/keys.rs:43-56`) and the mode is asserted by `secret_key_round_trip_and_mode`. (The *containing directory* is 0755 — see S4 — but the key files themselves are correct.)
- **Private keys never enter the database, `db export`, or the tape.** The `encryption_keys` schema stores only `alias`, `fingerprint`, `public_key` (`src/db/migrations/001_initial.sql:23-34`); `insert_key` writes only those (`src/db/queries.rs:115-130`). `db export` emits table row *counts* (`src/main.rs:200-220`) — no key material. Per §8.8 private keys are never written to tape, and the implementation writes no `catalog.db` to envelopes at all yet (prior audit physical-MED), so there is not even a latent path.
- **The audit-events table carries no secrets.** `events` rows store `entity_label`, `field`, `old_value`, `new_value` (`src/db/events.rs`); for key operations the label is the public alias and values are statuses/fingerprints — never secret material (`src/cli/key.rs:97-188`, `src/tenant/mod.rs:36-62`).
- **No secrets in argv, environment, or logs.** tapectl never accepts key material on the command line (keys are generated internally; `key import` takes a *public*-key file path; `key export` prints only the public key). `RESTORE.sh` passes `age -i KEYFILE` a *path*, not the key. `tracing` output logs sizes, counts, unit names, and `dar_command` (paths), but no key bytes or passphrases (grepped across `src/`).
- **Cross-tenant crypto isolation holds.** Slices and envelopes are encrypted to tenant-key(s) + operator-key(s) (`src/staging/mod.rs:191-200`, `src/volume/write.rs:295-321`); a tenant can decrypt only files listing their recipient stanza, and the operator can decrypt all — the §2.1 model. Proven by `tests/tenant_isolation.rs` (independent-recipient decrypt, cross-decrypt rejection) and by `age`'s per-recipient header stanzas. (The failure *mode* where a keyless tenant silently gets operator-only encryption is prior audit **H13**, cross-referenced under S1.)
- **The two `unsafe` blocks are sound.** `src/tape/ioctl.rs:88` and `:133` are `nix::libc::ioctl` FFI calls with `#[repr(C)]` `MtOp`/`MtGet` structs whose layouts match `<linux/mtio.h>` on LP64; pointers are valid for the call, and the return code is checked before use. No `unsafe` appears elsewhere in `src/` (grep-verified).
- **Supply-chain fundamentals.** `age` pinned 0.11.2 (§2.16), bundled SQLite 3.51.3 (current), zero git dependencies (see S5 for the one reachability-gated advisory).
